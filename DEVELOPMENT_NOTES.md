# Development Notes

This project is the reference implementation for building GiveWP/Give payment gateway add-ons.

Use these notes as a checklist when creating another gateway, for example `give-payu-gateway`.

## Project Shape

- Keep each payment provider in a separate plugin and repository.
- Use a provider-specific plugin slug, text domain, option name, REST namespace and asset names.
- Keep the Give gateway ID short and stable, for example `przelewy24` or `payu`.
- Include `README.md`, `CHANGELOG.md`, `languages/*.pot`, `languages/*-pl_PL.po` and compiled `.mo`.
- Build release ZIP files with the plugin directory as the top-level folder.

## Give Integration

- Require the Give plugin with the WordPress `Requires Plugins: give` header.
- Block activation if Give is not active.
- Register Give hooks only when Give is active.
- Add a dedicated settings section under:

```text
Donations > Settings > Payment Gateways
```

- Add a plugin list `Settings` action linking directly to the gateway settings.
- Support Visual Donation Form Builder via the Give payment gateway class.
- Keep the legacy form field markup for compatibility when practical.

## Settings

- Provide sandbox and production modes.
- Store settings in one provider-specific option array.
- Preserve masked secrets on save.
- Show masked API/secret fields as `***` after save.
- Add required field markers for required credentials.
- Add a "test API access" action where the provider API supports it.
- Keep source strings in English and add Polish translations.

## Payment Flow

- Register offsite transactions through the provider API.
- Generate a unique provider session ID containing the Give donation ID.
- Store the session ID in donation meta before redirecting the donor.
- Use the Give success page as the return URL.
- Use a provider-specific REST endpoint as the status/webhook URL.
- Use the Give form title in the payment description when available.
- Keep a fallback description using the donation ID.

## Webhook Security

- Verify provider webhook signatures before doing any state changes.
- Parse the donation ID only from a session ID format generated by the plugin.
- Confirm the webhook session ID matches the session ID stored on the donation.
- Confirm amount and currency match the donation.
- Confirm provider merchant/account identifiers match configured credentials.
- Verify the transaction through the provider API before completing the donation.
- Make webhook handling idempotent.
- Use a per-donation processing lock to avoid duplicate processing during concurrent callbacks.
- Add stale-lock recovery so a failed process does not permanently block a donation.
- Return `200 ok` for safe duplicate callbacks after a donation is already complete.

## Donation Updates

- Complete the Give donation only after provider-side transaction verification succeeds.
- Store useful provider identifiers in donation meta.
- Add a private donation note with the visible provider transaction ID when available.
- Avoid exposing internal API identifiers in admin notes unless they help support.

## Logging

- Use Give's native log model when available.
- Use category `Payment`.
- Use the provider name as source, for example `Przelewy24` or `PayU`.
- Log successful registration and verification as `success`.
- Log rejected or failed webhook/API states as `error`.
- Log duplicate callbacks as `warning`.
- Include structured context, but never log raw secrets.

## Release Checklist

- Run PHP syntax checks.
- Test API access in sandbox.
- Run a full sandbox payment on a public HTTPS WordPress site.
- Confirm status changes from `Pending` to `Complete`.
- Confirm provider transaction ID appears in the donation note.
- Confirm Give logs have expected type/category/source.
- Build the release ZIP.
- Commit changes.
- Tag the release.
- Create a GitHub Release and upload the ZIP.
- Before final `1.0.0`, run one small live production payment after the provider account/domain is fully configured.
